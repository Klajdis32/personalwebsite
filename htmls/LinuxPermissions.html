  • Activity 1

a) Edit the file /etc/login.defs accordingly so that there is an expiration warning of 30 days as well as a minimum retention period of 2 days.

b) Edit the file /etc/security/pwquality.conf accordingly so that the maximum number of failed password attempts during user login is 4.

Solution:

a) To set an expiration warning of 30 days as well as a minimum retention period of 2 days, follow these steps:

  1. Open the Terminal

  2. Open the file /etc/login.defs with administrator privileges:

    sudo nano /etc/login.defs

  3. Change the fields PASS_WARN_AGE and PASS_MIN_DAYS to 30 and 2, respectively

  4. Save the file: Ctrl+O and then Enter



b) To set the maximum number of failed password attempts during user login to 4, follow these steps:


  1. Press Ctrl+X to return to the terminal

  2. Open the file /etc/security/pwquality.conf with administrator privileges:

    sudo nano /etc/security/pwquality.conf

  3. Change the field retry to 4

  4. Save the file: Ctrl+O and then Enter

      Solution:


      Having logged in as user1, execute the following command in the terminal:

      cd /usr/dept1


      and observe that the folder opens normally. However, when executing the command:

    • Activity 3

      Log out from your account and log in as user1. Try to open the TextEditor application using the terminal by typing the path for the executable file of TextEditor: /usr/bin/gnome-text-editor

      What do you observe and why?

      Solution:


      After logging out from the current account and logging in as user1, we open the terminal and attempt to run the Text Editor application using the command:

      /usr/bin/gnome-text-editor

      We receive the error message: Permission denied



      This happens because previously, as root, we removed the execution permission of the file using the command:

      sudo chmod o-x /usr/bin/gnome-text-editor


    • Activity 4

Grant permissions as root to the group that user1 belongs to so that both the group and user1 can use the TextEditor application. Then, confirm the ability to execute the TextEditor application by logging in as user1.

Solution:


Having logged in as admin, open the terminal and follow the steps below to grant permissions to user1's group so they can use the TextEditor application:

  1. Change the group ownership of the executable file so that it belongs to the dept1 group, which includes user1. This is done by executing the command that sets the file’s group to dept1, while the owner remains root:

    sudo chown root:dept1 /usr/bin/gnome-text-editor


  2. Grant execution permissions to the dept1 group so that users in the group can use the file. Set the permissions so that root has full access (rwx - read, write, execute), the dept1 group can read and execute the file (r-x), while other users have no access.

    sudo chmod 750 /usr/bin/gnome-text-editor

  3. Verify the changes by checking the file permissions with the command:

    ls -l /usr/bin/gnome-text-editor

  4. Finally, log in as user1 and try to execute the TextEditor application by typing /usr/bin/gnome-text-editor in the terminal. The application opens successfully!


    • Activity 5

Modify the permissions of the corresponding directories and text files you created for the dept1 and dept2 groups so that all system users have full permissions on the file file2.txt, while for file1.txt, full permissions should be granted only to users belonging to the dept1 group, and all other system users should have only read and execute permissions.

Solution:


Having logged in as admin, open the terminal and follow the steps below to correctly set the file permissions as per the instructions:

  1. Grant full permissions (rwx) to the file /usr/dept2/file2.txt for all users:

    sudo chmod 777 /usr/dept2/file2.txt


    This allows all system users to read, write, and execute the file.

  2. Change the group ownership of the file /usr/dept1/file1.txt so that it belongs to the dept1 group:

    sudo chown root:dept1 /usr/dept1/file1.txt


    The root remains the owner, but now the group that has access to the file is dept1.

  3. Set the permissions on file1.txt:

    sudo chmod 774 /usr/dept1/file1.txt


    This grants full permissions (rwx) to the users of the dept1 group and only read (r--) permissions for all other users.


  4. To verify that everything is set correctly, execute:

    sudo ls -l /usr/dept1/file1.txt sudo ls -l /usr/dept2/file2.txt



    • Activity 6

  1. Using the root account, add the user useradmin to the syslog group (which is a system user) so that they can access the log files.

  2. Add the user useradmin to the users who can execute the sudo command using the command sudo adduser useradmin sudo

  3. Log out from root and enter the wrong password three times when attempting to log in as user1.

  4. Log in with the useradmin account and use the command sudo grep “authentication failure”

/var/log/auth.log to record the failed login attempts of user1.

Solution:


  1. To allow the user useradmin to access the log files, we need to add them to the syslog group. This is done by executing the command:

    sudo usermod -aG syslog useradmin


    To verify that the user was successfully added to the group, execute:

    groups useradmin


  2. To allow the user useradmin to execute commands with sudo, we need to add them to the sudo group. This is done by executing the command:

sudo adduser useradmin sudo


To verify that the user was successfully added to the group, execute:

groups useradmin
